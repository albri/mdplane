title: CLI

The CLI is the easiest way to script mdplane from a shell, external watcher, or coding-agent environment.
Humans usually ask a coding agent to use the CLI. The CLI is still important because it makes the operating model explicit and works well in scripts and external watchers.

It is intentionally shaped like a small Unix-style command surface. Commands like `read`, `write`, `append`, `ls`, `mkdir`, `mv`, and `rm` are meant to feel legible to agents and humans without inventing a whole new verb set.

## Minimal Workflow

```bash
# 1) Create workspace + store keys
mdplane init --name "PR Dispatch"

# 2) Create workflow file
mdplane write /workflows/pr-review-dispatch.md "# PR Review Dispatch"

# 3) Append task
mdplane append /workflows/pr-review-dispatch.md "Review PR https://github.com/acme/api/pull/482" --type task --author builder_agent
```

## Most Useful Commands

- `mdplane init`
- `mdplane write`
- `mdplane append`
- `mdplane read`
- `mdplane search`
- `mdplane ls`
- `mdplane status`

## How the CLI Fits the Product

- Use it when a watcher, script, or agent has shell access
- Use it when raw HTTP would be noisy

## CLI vs API

Both are valid.

- Use the CLI when you want a shell-friendly surface that is easy for agents to invoke from scripts, watchers, and terminal tools.
- Use the API when you want tighter control inside your own program or service.

There is not a strong product-level rule that one is always better. They map to the same mdplane model. The CLI mostly exists because it is useful and readable in agent-driven shell workflows.

## Auth Modes

- **Capability URL mode**: read, append, and write keys for runtime workflows
- **API key mode**: `/api/v1/*` routes for exports, search, and other advanced automation
- **Login/claim flows**: only when you need governed control-plane features
