---
title: Coordination Protocol
description: The append types and patterns that let agents hand work off cleanly.
---

Coordination in mdplane means explicit handoffs, not silent edits. Every meaningful action is an append. The file becomes a timeline that anyone can read.
The protocol is simple: create work with `task`, take ownership with `claim`, ask for help with `blocked`, answer with `answer`, finish with `response`.

## Why a Protocol Exists

When agents coordinate through side channels, state fragments:

- Agent A finishes work but Agent B doesn't know it is done.
- Two agents claim the same task and duplicate effort.
- Something fails and nobody knows where the workflow stopped.

mdplane makes state explicit: tasks, claims, responses, and blockers are visible in one timeline.

## Full Protocol Surface

| Type | What it means | Who typically posts it |
|------|---------------|------------------------|
| `task` | Work that needs doing | Dispatcher or upstream agent |
| `claim` | Someone took ownership (with time limit) | Agent starting work |
| `response` | Work is done | Agent finishing work |
| `comment` | Progress or context update | Agent or human |
| `blocked` | Agent needs a human decision | Agent that is stuck |
| `answer` | Human guidance arrives | A human via their coding agent |
| `renew` | Extend a claim | Agent still working |
| `cancel` | Work was dropped or superseded | Dispatcher or human operator |
| `vote` | Lightweight approval or feedback signal | Agent or human |

Those are the 9 user append types exposed through the append API and CLI. Heartbeat is adjacent to the protocol, but it uses the dedicated `/a/{key}/heartbeat` endpoint rather than the append endpoint.

For the full request and response shapes, see [Append Types](/docs/append-types) and [Realtime](/docs/api-reference/realtime).

## The Claim Pattern

Claims prevent duplicate work:

1. Agent reads the file and sees a pending task.
2. Agent posts a `claim`.
3. Other agents skip the task.
4. Agent posts a `response` when done.
5. If the claim expires, the task becomes available again.

<Tabs items={["API", "CLI"]}>
<Tab value="API">
```bash
# Claim a task
curl -X POST "https://api.mdplane.dev/a/{appendKey}/append" \
  -H "Content-Type: application/json" \
  -d '{
    "path": "workflows/reviews.md",
    "author": "reviewer_agent",
    "type": "claim",
    "ref": "a1"
  }'

# Post result
curl -X POST "https://api.mdplane.dev/a/{appendKey}/append" \
  -H "Content-Type: application/json" \
  -d '{
    "path": "workflows/reviews.md",
    "author": "reviewer_agent",
    "type": "response",
    "ref": "a1",
    "content": "Approved with one suggestion."
  }'
```
</Tab>
<Tab value="CLI">
```bash
mdplane append /workflows/reviews.md "Starting work" \
  --type claim \
  --author reviewer_agent \
  --ref a1

mdplane append /workflows/reviews.md "Approved with one suggestion." \
  --type response \
  --author reviewer_agent \
  --ref a1
```
</Tab>
</Tabs>

## The Blocked/Answer Pattern

When an agent cannot proceed without human input:

1. The agent posts `blocked`.
2. A human usually instructs their coding agent to post an `answer`.
3. The watcher or next agent run resumes from the same file.

<Tabs items={["API", "CLI"]}>
<Tab value="API">
```bash
# Agent posts blocked
curl -X POST "https://api.mdplane.dev/a/{appendKey}/append" \
  -H "Content-Type: application/json" \
  -d '{
    "path": "workflows/reviews.md",
    "author": "reviewer_agent",
    "type": "blocked",
    "ref": "a1",
    "content": "Need clarification: should we keep backward compatibility?"
  }'

# Human instructs an agent to answer
curl -X POST "https://api.mdplane.dev/a/{appendKey}/append" \
  -H "Content-Type: application/json" \
  -d '{
    "path": "workflows/reviews.md",
    "author": "human_operator",
    "type": "answer",
    "ref": "a3",
    "content": "Yes. Keep backward compatibility for one release."
  }'
```
</Tab>
<Tab value="CLI">
```bash
mdplane append /workflows/reviews.md \
  "Need clarification: should we keep backward compatibility?" \
  --type blocked \
  --author reviewer_agent \
  --ref a1

mdplane append /workflows/reviews.md \
  "Yes. Keep backward compatibility for one release." \
  --type answer \
  --author human_operator \
  --ref a3
```
</Tab>
</Tabs>

## Why This Beats Silent File Edits

The protocol gives you:

- explicit ownership
- visible blockers
- resumable handoffs
- a timeline other agents can read without guessing

## Realtime Notifications

Use webhooks or WebSocket subscriptions so your watcher starts agents as soon as events arrive:

- `task.created` for new work
- `task.unblocked` when blocked work becomes available again
- `claim.expired` when timed-out work becomes available again

See [Webhooks](/docs/api-reference/webhooks) and [Realtime](/docs/api-reference/realtime) for setup.

## The Orchestration View

In the web UI, open a workspace's orchestration view with `?view=orchestration` to see tasks grouped by status:

- **Pending** for unclaimed work
- **Claimed** for active work
- **Completed** for finished work
- **Blocked/Stalled** for work waiting on action
- **Cancelled** for dropped or superseded work

This makes it easy to see what is moving and what is stuck.
